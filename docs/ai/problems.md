# Known Problems & Issues

> Phase 1 Audit — June 2026

## Issue Severity Guide

- 🔴 **CRITICAL**: System produces incorrect results. Must fix before scaling.
- 🟠 **HIGH**: Significant quality degradation. Fix in near-term.
- 🟡 **MEDIUM**: Quality and reliability concerns. Fix when redesigning.
- 🟢 **LOW**: Minor improvements. Fix opportunistically.

---

## 🔴 CRITICAL Issues

### C1: Clustering Relies Solely on Embedding Similarity

**Location**: `clustering_service.py:300-313`  
**Impact**: Unrelated events with similar language are merged into the same story.

Cosine similarity ≥ 0.80 on `title + description` embeddings is the ONLY signal used to decide whether two articles describe the same event. No event type, actor, location, or time matching is performed.

**Example failures**:
- Two different elections in the same country → merged
- Two different arrests of different people → merged
- Weather events in different cities → merged

### C2: No Event Extraction

**Location**: Entire pipeline  
**Impact**: The system cannot distinguish between different types of events.

The pipeline jumps directly from raw article text to LLM summary. There is no intermediate event structure (`eventType`, `actors`, `targets`, `location`, `eventTime`). The LLM prompt assumes all articles in a cluster describe "a single news event" without verification.

### C3: NER Misclassifies Entities

**Location**: `ner_service.py:118-127`, `ner_service.py:98-99`  
**Impact**: Incorrect entity types displayed to users, incorrect tagging.

The `en_core_web_sm` model (12MB) is too small for news domain NER. The regex fallback defaults ALL unknown capitalized phrases to PERSON.

**Example failures**:
- "MoU" → PERSON ❌ (should be AGREEMENT)
- "Andhra Pradesh" → PERSON ❌ (should be STATE)
- "Birzeit University" → PERSON ❌ (should be ORG)
- "Supreme Court" → PERSON ❌ (should be ORG)
- "Green Energy" → PERSON ❌ (should be TECHNOLOGY or concept)

### C4: Contradictions Are Hallucinated

**Location**: `ai_service.py:174-209`  
**Impact**: Users see fabricated contradictions between sources.

Contradictions are detected by a single LLM call that simultaneously generates the summary, timeline, AND source differences. The LLM has no structured data to compare — it operates on raw text and tends to invent differences to fill the required schema fields.

### C5: Summary Before Understanding

**Location**: `clustering_service.py:61-213`  
**Impact**: Summaries are generated from raw text without structured knowledge.

The pipeline is: cluster → LLM prompt with raw text → save everything. There is no:
- Event extraction before clustering
- Entity canonicalization before summarization
- Knowledge graph construction before analysis
- Fact extraction before contradiction detection

## 🟠 HIGH Issues

### H1: Publication Time Conflated with Event Time

**Location**: `ai_service.py:181-183`  
**Impact**: Same event from different time zones appears as different events.

Only `published_at` from RSS/GNews is available. No `eventTime` is extracted from article text. The LLM prompt says "Published: {time}" which misleads the model into thinking articles published at different times describe different events.

### H2: HDBSCAN Parameters Too Loose

**Location**: `clustering_service.py:368-373`  
**Impact**: Over-merging of topically similar but distinct events.

`cluster_selection_epsilon=0.35` with `min_cluster_size=2` on embedding-only features is too permissive. Articles that share topic vocabulary but describe different events will be clustered together.

### H3: Embedding Excludes Article Body

**Location**: `tasks.py:138`  
**Impact**: Articles with similar titles but different bodies cluster together.

`text_to_embed = f"{article.title or ''} {article.description or ''}"` — the full article content is NOT embedded. This means clustering decisions are based on ~200 chars instead of the full article text.

### H4: No Entity Canonicalization

**Location**: Pipeline-wide  
**Impact**: Same entity appears as multiple different entries.

"Rahul Gandhi", "Mr Gandhi", "Congress leader Rahul Gandhi" are stored as three separate entities with no linking. This breaks entity-based story comparison and search.

### H5: Difference Engine Is LLM-Only

**Location**: `ai_service.py:95-105`  
**Impact**: Source differences are unreliable and often fabricated.

Source differences are generated by a single LLM call, not by structured comparison of extracted facts. The LLM only sees the first 3000 chars of each article, so it misses content differences in longer articles.

## 🟡 MEDIUM Issues

### M1: Regex Fallback Defaults to PERSON

**Location**: `ner_service.py:98-99`  
Any capitalized phrase not in the tiny hardcoded org/location dictionaries defaults to PERSON.

### M2: Article Content Truncated to 3000 chars

**Location**: `ai_service.py:182`  
Only first 3000 chars of article content sent to LLM. Long articles lose critical details.

### M3: Timeline Dates Are Unparseable

**Location**: `clustering_service.py:136-140`  
LLM returns strings like "08:00 AM UTC" which fail ISO parsing. Cannot be sorted or compared.

### M4: No Story Splitting Mechanism

**Location**: `clustering_service.py`  
Once merged, articles can never be separated. Incorrect clusters are permanent.

### M5: Source Coverage Uses LLM String Splitting

**Location**: `clustering_service.py:163-166`  
`focus_area` is extracted by splitting `unique_information` on `.` — fragile and incomplete.

---

## 🟢 LOW Issues

### L1: Arbitrary Entity Limit of 15

**Location**: `clustering_service.py:190`  
High-entity stories lose important context.

### L2: Tag Limit of 5

**Location**: `clustering_service.py:207`  
Only 5 tags per story limits search discoverability.

### L3: No Confidence Scores

**Location**: Pipeline-wide  
No clustering confidence, entity confidence, or contradiction confidence is reported.
